# aube bug reproductions

Minimal test cases for bugs found migrating a large monorepo from pnpm to aube.
Each directory is self-contained: `cd` into it and run the commands in its README.

GitHub Actions workflows run each repro on push, comparing aube's behavior against pnpm.

## All fixed in beta.10

All three bugs below were fixed in beta.10. The repros and CI workflows remain as
regression tests.

1. **`alias-override-conflict/`** — _Fixed in beta.10._ `npm:` alias replaced by an
   unrelated `pnpm.overrides` entry.

2. **`overrides-not-read-from-workspace/`** — _Fixed in beta.10._ `overrides` in
   `pnpm-workspace.yaml` ignored by `--frozen-lockfile`.

3. **`pnpm-lockfile-host-only-platforms/`** — _Fixed in beta.10._ `pnpm-lock.yaml`
   generated by aube only contained optional deps for the host platform.

## Observations (not bugs, but migration friction)

**Transitive dependency version drift during re-resolution.** When aube re-resolves a
lockfile (e.g. because overrides changed), it picks newer versions of transitive dependencies
than pnpm had pinned. This is technically correct behavior — aube resolves fresh while pnpm's
incremental lockfile preserves older pinned versions. It causes friction for migrations since
packages like `@types/hapi__shot` (deprecated empty stub at latest vs working older version),
`hls-parser`, and `@react-aria/virtualizer` resolve to newer versions with breaking type
changes. Workaround: pin affected transitive dependencies via `overrides` or `catalog`.
