# aube bug reproductions

Minimal test cases for bugs found migrating a large monorepo from pnpm to aube.
Each directory is self-contained: `cd` into it and run the commands in its README.

GitHub Actions workflows run each repro on push, comparing aube's behavior against pnpm.

## Fixed in beta.10

Bugs 1-3 were fixed in beta.10. The repros and CI workflows remain as regression tests.

1. **`alias-override-conflict/`** — _Fixed in beta.10._ `npm:` alias replaced by an
   unrelated `pnpm.overrides` entry.

2. **`overrides-not-read-from-workspace/`** — _Fixed in beta.10._ `overrides` in
   `pnpm-workspace.yaml` ignored by `--frozen-lockfile`.

3. **`pnpm-lockfile-host-only-platforms/`** — _Fixed in beta.10._ `pnpm-lock.yaml`
   generated by aube only contained optional deps for the host platform.

## Open issues (as of beta.10)

4. **`temporal-duration-type/`** — `@temporalio/common` depends on `ms@3.0.0-canary.1`
   which exports a strict template literal `StringValue` type. With pnpm, `ms@2.x` gets
   hoisted from other packages and TypeScript resolves `Duration` through it (where
   `StringValue` is plain `string`). aube's strict isolation correctly gives each package
   its declared version, but this breaks code that passes string literals like `'1s'` to
   temporal APIs.

5. **`openai-version-conflict/`** — When two workspace packages depend on different
   major versions of `openai` (e.g. `^4.47.0` and `^6.0.0`), aube's virtual store
   gives each its own copy with distinct type identities. TypeScript treats
   `ChatCompletionMessageParam` from `openai@4.47.1` as incompatible with the same
   type from `openai@6.34.0`, even though they're structurally identical. pnpm's
   hoisting avoids this by deduplicating to a single type identity where possible.

6. **`type-augmentation-not-visible/`** — `declare module '@hapi/hapi'` augmentations
   defined in one workspace package are not visible in another package that depends on
   it. With pnpm, both packages resolve `@hapi/hapi` to the same hoisted copy, so the
   augmentation merges correctly. With aube's virtual store, each package gets its own
   isolated `@hapi/hapi`, so the augmentation targets a different module identity.
